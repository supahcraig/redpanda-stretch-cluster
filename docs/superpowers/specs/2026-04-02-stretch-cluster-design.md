# Stretch Redpanda Cluster on AWS — Design Spec

**Date:** 2026-04-02  
**Status:** Approved

---

## Overview

Terraform and Ansible automation to deploy a "stretch" Redpanda cluster spanning 3 AWS regions, with brokers distributed across unique Availability Zones for fault-domain isolation. The cluster is designed for OMB (OpenMessaging Benchmark) performance testing, with rack-aware replica placement and explicit leader affinity to a preferred region.

Azure and GCP support will follow the same structural pattern in a future phase. The AWS implementation serves as the reference.

---

## Topology

- **5 brokers total**: 2 regions with 2 brokers each (one per AZ), 1 region with 1 broker
- **Every broker is in its own AZ** — AZ is the rack/fault domain
- **Default regions**: `us-east-1` (2 brokers), `us-east-2` (2 brokers), `us-west-2` (1 broker)
- All regions/counts are overridable via `terraform.tfvars`
- Replication factor 5 by default (all brokers hold every partition replica)

## Terraform: Variable Design

### Regions variable

Provider aliases cannot be dynamically generated in Terraform, so the number of region slots is fixed at 3 (matching the stretch topology). Region names, broker counts, AZs, VPC CIDRs, and disk config are all variable-driven:

```hcl
variable "regions" {
  type = list(object({
    name         = string
    broker_count = number
    azs          = list(string)
    vpc_cidr     = string
  }))
  default = [
    { name = "us-east-1", broker_count = 2, azs = ["us-east-1a", "us-east-1b"], vpc_cidr = "10.0.0.0/16" },
    { name = "us-east-2", broker_count = 2, azs = ["us-east-2a", "us-east-2b"], vpc_cidr = "10.1.0.0/16" },
    { name = "us-west-2", broker_count = 1, azs = ["us-west-2a"],               vpc_cidr = "10.2.0.0/16" },
  ]
}
```

### Other key variables

```hcl
variable "deployment_prefix" {
  default = "rp-stretch"
}

variable "ssh_key_name" {
  default = "redpanda-stretch-cluster"
  # The same key name is registered in all 3 regions with the same public key material.
}

variable "public_key_path" {
  default = "~/.ssh/id_rsa.pub"
}

variable "broker_instance_type" {
  default = "m7gd.2xlarge"
  # m7gd = Graviton3 ARM + local NVMe. Use m7g.2xlarge + disk_type="ebs" in regions
  # where m7gd is unavailable.
}

variable "machine_architecture" {
  default = "arm64"
  # Must match broker_instance_type. m7gd -> arm64. x86 instances -> x86_64.
}

variable "disk_type" {
  default = "instance_store"
  # "instance_store" uses local NVMe (m7gd, i3, i4i, etc.)
  # "ebs" provisions an aws_ebs_volume per broker (use with non-d instance types)
}

variable "ebs_volume_size_gb" {
  default = 500
}

variable "ebs_volume_type" {
  default = "gp3"
}

variable "ebs_iops" {
  default = 16000
}

variable "leader_preference" {
  type    = list(string)
  default = ["us-east-1", "us-east-2", "us-west-2"]
  # Ordered from most to least preferred region for partition leadership.
  # Must reference region names defined in var.regions.
}

variable "redpanda_version" {
  default = "latest"
}
```

## Terraform: Provider Aliases

Three provider aliases with fixed names (`region0`, `region1`, `region2`) pull their actual region from `var.regions[*].name`:

```hcl
provider "aws" { alias = "region0"; region = var.regions[0].name }
provider "aws" { alias = "region1"; region = var.regions[1].name }
provider "aws" { alias = "region2"; region = var.regions[2].name }
```

Each regional module is instantiated once per slot, passing its provider alias via `providers = { aws = aws.region0 }`.

## Terraform: Regional Module

Each instantiation of `modules/aws/regional/` creates:

### SSH Key Pair
- `aws_key_pair` using `var.public_key_path`
- Name: `${var.ssh_key_name}` — same name and same public key material in all 3 regions, so a single `~/.ssh/` entry works everywhere

### VPC & Networking
- VPC with the supplied CIDR, DNS hostnames enabled
- One public subnet per AZ in `var.azs` (`map_public_ip_on_launch = true`)
- Internet Gateway + default route table entry
- Outputs: `vpc_id`, `route_table_id`, `subnet_ids` (map of AZ → subnet_id)

### Security Group
| Port | Protocol | Source | Purpose |
|------|----------|--------|---------|
| 22 | TCP | 0.0.0.0/0 | SSH / Ansible |
| 9092 | TCP | 0.0.0.0/0 | Kafka (clients) |
| 9644 | TCP | 0.0.0.0/0 | Admin API |
| 8082 | TCP | 0.0.0.0/0 | HTTP Proxy (Pandaproxy) |
| 33145 | TCP | all 3 VPC CIDRs | Internal RPC (inter-broker only) |
| All | All | own VPC CIDR | Intra-VPC traffic |

The 3 VPC CIDRs are passed into the module as `var.peer_cidrs`.

### EC2 Instances
- `broker_count` instances distributed across AZs: `element(var.azs, count.index)`
- Public IP assigned (`associate_public_ip_addr = true`)
- AMI: latest Ubuntu 22.04 for `var.machine_architecture` (looked up via `data "aws_ami"`)
- Instance type: `var.broker_instance_type`

**Disk — instance_store path** (`var.disk_type == "instance_store"`):
- `ephemeral_block_device` blocks configured to surface local NVMe to the OS
- Device names: `/dev/nvme0n1` (single-NVMe types like `m7gd`), `/dev/nvme0n1` + `/dev/nvme1n1` (dual-NVMe types like `i3en`)
- Ansible handles format (XFS) + mount at `/var/lib/redpanda/data`; RAID-0 via `mrlesmithjr.mdadm` when 2 drives are present

**Disk — EBS path** (`var.disk_type == "ebs"`):
- `aws_ebs_volume` per broker: type `var.ebs_volume_type`, size `var.ebs_volume_size_gb`, IOPS `var.ebs_iops`
- `aws_volume_attachment` attaches to `/dev/sdf` (presents as `/dev/nvme1n1` on Nitro)
- Ansible handles format (XFS) + mount at `/var/lib/redpanda/data`

**Module outputs per broker**: `{ name, public_ip, private_ip, az, region }`

## Terraform: VPC Peering (Full Mesh)

Three peering connections managed in `terraform/aws/main.tf`:

| Connection | Requester | Accepter |
|------------|-----------|----------|
| 0↔1 | `aws.region0` | `aws.region1` |
| 0↔2 | `aws.region0` | `aws.region2` |
| 1↔2 | `aws.region1` | `aws.region2` |

Each connection requires:
1. `aws_vpc_peering_connection` in the requester
2. `aws_vpc_peering_connection_accepter` in the accepter (different provider alias)
3. Route table entries in **both** VPCs for each peer's CIDR via the peering connection ID

The regional module exposes `vpc_id` and `route_table_id` as outputs for the root to wire up routes.

## Terraform: hosts.ini Generation

A `local_file` resource at the root uses `templatefile()` to render `ansible/hosts.ini` from all broker outputs. Format:

```ini
[redpanda]
broker-0-us-east-1 ansible_host=<public_ip> private_ip=<private_ip> rack=us-east-1a ansible_user=ubuntu
broker-1-us-east-1 ansible_host=<public_ip> private_ip=<private_ip> rack=us-east-1b ansible_user=ubuntu
broker-0-us-east-2 ansible_host=<public_ip> private_ip=<private_ip> rack=us-east-2a ansible_user=ubuntu
broker-1-us-east-2 ansible_host=<public_ip> private_ip=<private_ip> rack=us-east-2b ansible_user=ubuntu
broker-0-us-west-2 ansible_host=<public_ip> private_ip=<private_ip> rack=us-west-2a ansible_user=ubuntu

[all:vars]
ansible_ssh_private_key_file=~/.ssh/id_rsa
redpanda_version=<var.redpanda_version>
```

`rack` = the broker's AZ name (the fault domain for replica placement).

`hosts.ini` is gitignored — it contains live IP addresses and is regenerated on every `terraform apply`.

## Ansible: Cluster Provisioning

### Playbook
`ansible/provision-cluster.yml` — adapted from `deployment-automation/ansible/provision-cluster.yml`.

Core roles remain the same:
- `redpanda.cluster.system_setup` — prereqs, sysctl, disk format/mount
- `redpanda.cluster.sysctl_setup`
- `redpanda.cluster.redpanda_broker` — install, configure, start Redpanda

### Key configuration changes from single-region

**Seeds**: All 5 brokers' private IPs are passed as seeds. Inter-region private IPs are reachable via VPC peering. This is the primary difference from the single-region playbook.

**Advertised listeners**: `advertise_public_ips: true` — clients connect via public IPs. Internal RPC uses private IPs automatically.

**Rack awareness** (per-broker): The `rack` host variable from `hosts.ini` is passed to the broker role to set `redpanda.rack` in `redpanda.yaml`.

**Cluster-wide properties** set via the broker role:

```yaml
redpanda_cluster_properties:
  enable_rack_awareness: true
  default_topic_replications: 5
  internal_topic_replication_factor: 5
  default_leaders_preference:
    racks: <AZs expanded from leader_preference, e.g. ["us-east-1a","us-east-1b","us-east-2a","us-east-2b","us-west-2a"]>
```

The `default_leaders_preference` racks list is computed in Terraform: for each region in `var.leader_preference` (in order), its AZs are looked up from `var.regions` and appended to the list. The resulting ordered AZ list is written into `hosts.ini` as an `[all:vars]` entry (`redpanda_leader_rack_preference`) so the Ansible playbook can pass it directly to the broker role without any further logic. This gives region-level leader preference semantics while operating at AZ-level racks.

### Disk setup
The `system_setup` role handles NVMe formatting (XFS) and mounting at `/var/lib/redpanda/data`.
- Single NVMe (`m7gd`, `i4i`): format + mount directly
- Dual NVMe (`i3en`): RAID-0 via `mrlesmithjr.mdadm`, then format + mount
- EBS: same format + mount path; device presented as NVMe on Nitro instances

## Workflow

```
# 1. Initialize
cd terraform/aws
terraform init

# 2. Review / customize
cp terraform.tfvars.example terraform.tfvars
# edit regions, instance type, leader_preference, etc.

# 3. Provision infrastructure
terraform apply
# generates ansible/hosts.ini

# 4. Install Ansible dependencies (one-time)
export ANSIBLE_COLLECTIONS_PATHS=../../artifacts/collections
export ANSIBLE_ROLES_PATH=../../artifacts/roles
ansible-galaxy collection install -r ../../requirements.yml --force -p $ANSIBLE_COLLECTIONS_PATHS
ansible-galaxy role install -r ../../requirements.yml --force -p $ANSIBLE_ROLES_PATH

# 5. Bootstrap Redpanda
export ANSIBLE_INVENTORY=../../ansible/hosts.ini
ansible-playbook ../../ansible/provision-cluster.yml --private-key ~/.ssh/id_rsa

# 6. Tear down
terraform destroy
```

## Open Questions / Future Work

- Monitoring (`deploy-monitor.yml`) and client nodes are not included in phase 1 — can be added following the same pattern as `deployment-automation`
- Azure and GCP implementations follow this spec as the reference; each gets its own `terraform/<cloud>/` root and `modules/<cloud>/regional/` module
- TLS is out of scope for now; the `provision-cluster-tls.yml` path from `deployment-automation` can be added later without structural changes
- Tiered Storage is out of scope for now; `provision-cluster-tiered-storage.yml` from `deployment-automation` can be added later
